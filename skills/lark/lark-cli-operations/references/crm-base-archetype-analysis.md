# CRM 底表类型识别与改造分析 SOP

**适用场景：** 完成 Base 全貌侦察（`lark-base-reconnaissance.md`）后，需要进一步判断该 Base 属于哪种 CRM 类型，判断是否适配有膜有漾业务，以及输出改造方案。

**前置条件：** 已完成 `lark-base-reconnaissance.md` 的八步侦察流程，拿到完整的表结构、字段、Workflow、仪表盘数据。

---

## 1. 两种 CRM 底表类型

基于有膜有漾业务场景，CRM 多维表格通常分为两种类型。拿到一个 Base 后，先判断它属于哪一种：

### 类型 A：门店运营型 CRM

**典型特征：**

| 维度 | 标志性信号 |
|------|-----------|
| 核心表 | 包含**工单表**、**施工记录表**、**报价表** |
| 状态机 | 工单状态包含**审核/施工/返工/完工确认**等施工交付环节 |
| 角色 | 含**师傅/店长/财务**等门店运营角色 |
| 产品 | **车膜/窗膜/改色膜**等门店服务项目 |
| 财务 | 含**财务核验表**，应收 vs 实收对账 |
| 售后 | 含**售后表**，记录返工/质保/投诉 |
| 客户 | **个体车主**维度，含车型/车牌/车辆照片 |

**适用于：** 门店日常运营管理（销售→报价→施工→交付→售后全链路）

**代表案例：** 第一个分析的有膜有漾 CRM（10表：员工角色/线索/报价/产品/客户档案/工单/施工/财务/售后/状态流转规则）

### 类型 B：销售管道型 CRM

**典型特征：**

| 维度 | 标志性信号 |
|------|-----------|
| 核心表 | 包含**商机管理**、**订单产品明细表** |
| 状态机 | 订单状态为**待处理/待发货/已完成**等交易物流环节 |
| 角色 | 含**销售经理**层级，按区域/门店管理 |
| 产品 | **通用产品类型**（如传感器/设备/软件） |
| 财务 | 含**毛利核算链**（lookup 售价→公式算毛利） |
| 客户 | **B2B 企业客户**维度，含行业/规模/分层 |
| 目标 | 含**销售目标管理**，按季度/人员分解 |

**适用于：** 多门店销售管理（市场活动→线索→商机→订单→客户分层），适合销售总监/市场部视角

**代表案例：** 第二个分析的通用 CRM 底表（11表：市场活动/线索/商机/订单/订单明细/客户信息/联系人/跟进/产品/销售目标/销售人员）

## 2. 快速识别清单

拿到一个未知 Base 后，快速扫描以下信号：

```bash
# 1. 看看有没有这些表名
lark-cli base +table-list --base-token <token> --as bot --format json

# 若包含以下任意 2 个 → 类型 A（门店运营型）：
#   工单表 / 施工记录表 / 报价表 / 财务核验表 / 售后表 / 客户车辆档案表

# 若包含以下任意 2 个 → 类型 B（销售管道型）：
#   商机管理 / 订单产品明细表 / 销售目标管理 / 市场活动管理 / 销售订单
```

**混合型识别：** 如果同时包含两类特征（如既有工单表又有商机表），则可能是**融合型**，需要在改造时注意模块边界。

## 3. 从当前 Base 到有膜有漾业务的改造分析

### 3.1 差距分析矩阵

对每个表，评估三个维度：

| 评估维度 | 分值 | 判断标准 |
|----------|------|----------|
| **字段适配度** | ⭐⭐⭐ | 字段直接可用或只需微调选项 |
|  | ⭐⭐ | 需要删改部分字段，替换类型 |
|  | ⭐ | 字段与业务完全不相关，需重建 |
| **关联适配度** | ⭐⭐⭐ | link 关系直接可用 |
|  | ⭐⭐ | 需调整关联目标表 |
|  | ⭐ | 需重新设计关联关系 |
| **流程适配度** | ⭐⭐⭐ | Workflow 直接可用或只需微调触发条件 |
|  | ⭐⭐ | 需修改 Workflow 步骤 |
|  | ⭐ | 需重新设计自动化流程 |

### 3.2 常见替换模式

**模式 A：字段选项替换（低风险）**

原 → 改：
```
线索来源：市场活动/官网咨询 → 抖音/小红书/门店到访/转介绍
丢单原因：客户采购计划调整/产品能力 → 预算不足/对比竞品/暂时不需要
订单状态：待处理/待发货/已完成 → 待施工/施工中/已完成/已取消
```

**模式 B：表结构整体替换（中风险）**

用新表替换旧表，保留关联接口：
```
产品管理（通用产品）         →  车膜项目产品表
客户信息 + 客户联系人（B2B） →  客户车辆档案表（个体车主）
```

**模式 C：模块新增（高风险）**

从门店运营型 CRM 导入完整模块：
```
新增：工单表（含8状态流转）
新增：施工记录表（含照片/返工）
新增：财务核验表（应收vs实收）
新增：售后表（返工/质保/投诉）
```

### 3.3 Workflow 启用策略

销售管道型 CRM 的 Workflow 通常**全部被禁用**（设计态而非运行态）。启用优先级：

| 优先级 | 流程 | 预估价值 |
|--------|------|----------|
| P0 | 线索分配通知 | ⭐⭐⭐ 销售日常必需 |
| P0 | 工单派工通知（如已导入） | ⭐⭐⭐ 施工流转必需 |
| P0 | 线索录入重复判断 | ⭐⭐⭐ 防重复销售 |
| P1 | 商机跟进提醒 | ⭐⭐ 防商机流失 |
| P1 | 线索自动分配 | ⭐⭐ 减少管理成本 |
| P2 | 产品上架/下架按钮 | ⭐ 锦上添花 |

## 4. 报告输出规范

完成 Base 分析后，输出两份文档：

### 文档 A：字段设计文档

**文件名：** `CRM底表字段设计文档.md`

**结构：**
```
# CRM 底表 — 表格设计字段文档

## 目录（11张表依次列出）

## 表1：表名
### 字段清单（表格：序号/字段名/类型/详细说明/选项值）
### 视图（如有）
### 字段总数

## ...

## 附录 A：跨表关联关系索引
## 附录 B：字段类型统计
## 附录 C：公式字段明细
```

**格式要求：**
- 每个 `select` 字段列出所有选项值（用 **加粗** 标注）
- 每个 `link` 字段标注目标表名
- 每个 `lookup` 字段标注来源表
- 附录 B 统计表使用占比分析

### 文档 B：改造方案

**文件名：** `CRM底表改造方案.md`

**结构：**
```
# CRM 底表改造方案 — 适配有膜有漾业务

## 一、改造原则
## 二、改造全景图
## 三、各表修改详细方案
### 表1：XXX — 修改类型（保留/微调/替换/新增）
#### 修改点表格（原字段 → 改为 → 原因）
## 四、Workflow 改造计划
## 五、仪表盘改造建议
## 六、改造成本评估
## 七、分阶段实施建议
## 八、Why Not Rebuild
```

**修改类型标记：**
| 标记 | 含义 |
|------|------|
| ✅ 保留微调 | 字段或选项值调整 |
| ⚠️ 中等修改 | 字段增删 + 关联调整 |
| ❌ 整体替换 | 用新表替换原表 |
| ➕ 新增导入 | 从其他 Base 引入新模块 |

## 5. 本次 Session 验证案例

本次 session 分析了两个 CRM Base：

**Base A — 门店运营型 CRM（有膜有漾专用）**
- 10 表 / 7 Workflow / 2 仪表盘
- 核心流程：线索→报价→工单→施工→财务→售后
- 适用角色：销售、师傅、店长、财务
- 完整状态机：25 条流转规则
- **结论：已适配有膜有漾业务，可直接使用**

**Base B — 销售管道型 CRM（通用模板）**
- 11 表 / 12 Workflow（全部禁用）/ 6 仪表盘
- 核心流程：市场活动→线索→商机→订单→客户分层
- 适用角色：销售经理、市场部、销售总监
- 毛利核算链：产品定价→lookup→公式折后价→自动算毛利
- **结论：需改造后使用，骨架保留，替换业务实体字段**
