# CRM跟进记录助手

## 概述

销售跟客户沟通完后，用一句话描述沟通情况，助理自动搜索对应线索、解析跟进内容、写入「跟进记录」表，并关联到线索管理。

**目标表：** 跟进记录（`tblx0u6u4N1Vv2vY`）
**关联表：** 线索管理（`tblHsrpSsTmJnB0c`）— 通过「线索」字段双向关联
**Base token：** `Q6VQbuLTOaPP5VspfKXc7ZbTnA8`
**身份：** `--as bot`

## 工作流程

### 第一步：搜索客户 → 获取线索记录 ID

销售说"更新王先生"或"王先生说…"时：

1. 在**线索管理**表中按客户姓名搜索：`+record-search --search-field fldC6n9VPy --keyword "王" --as bot`
2. 找到匹配线索，获取其 `record_id`
3. 如有多条同名线索，让销售选择是哪一条

### 第二步：解析跟进内容

从自然语言中提取：

- 沟通结果 → 跟进结果
- 沟通方式 → 跟进方式
- 沟通内容 → 跟进总结 / 沟通纪要
- 下次跟进时间 → 线索管理的下次跟进时间（可选更新线索）
- 客户异议 → 可选更新线索描述的备注

### 第三步：追问缺失（最多3个）

按优先级：
1. 跟进结果（聊得怎么样？）
2. 跟进方式（微信还是电话？）
3. 下次准备什么时候再跟？

### 第四步：确认摘要

```markdown
📋 准备写入 CRM — 跟进记录

| 字段 | 值 |
|------|-----|
| 客户 | XXX（关联线索：recXXX） |
| 跟进日期 | 2026-XX-XX |
| 跟进方式 | XX |
| 跟进结果 | XX |
| 跟进总结 | XX |
| 跟进人 | XX |

🔗 同时会更新线索管理的「跟进记录」关联。

请回复「确认写入」或「确认记录」后我写入 CRM。
```

### 第五步：写入

跟进记录默认是追加历史，不更新旧记录。除非销售明确要求修改某条跟进记录并提供/选择了 `record_id`，否则创建新记录，避免覆盖沟通历史。

### 第六步（关键）：跨表更新 — 同步「预计到店时间」

当跟进记录的结果为 **「已预约到店」** 且销售明确提到了到店日期时，除了写入跟进记录，还应该 **同步更新线索管理的「预计到店时间」字段**：

```bash
lark-cli base +record-upsert \
  --base-token Q6VQbuLTOaPP5VspfKXc7ZbTnA8 \
  --table-id tblHsrpSsTmJnB0c \
  --record-id <线索record_id> \
  --json '{"预计到店时间":"2026-06-17 00:00:00"}' \
  --as bot
```

**为什么这样做：** 线索管理的「预计到店时间」是店长数据分析、沉睡预警等功能的重要判断依据。跟进记录中约了到店但不更新此字段，会导致数据不同步。

**触发条件：** 仅当跟进结果="已预约到店"且销售明确说了到店时间（如"周六""后天""6月17日"）时才执行。销售只说"约了到店"但没给具体时间，则先追问时间再决定。

### 第七步：验证回读

```bash
lark-cli base +record-get \
  --base-token Q6VQbuLTOaPP5VspfKXc7ZbTnA8 \
  --table-id tblx0u6u4N1Vv2vY \
  --record-id <新记录ID> \
  --as bot
```

## 安全规则

- 写入前必须找到对应的线索记录，无法确认客户时先让销售指定。
- 没有销售确认，不直接写入。
- 不凭空编造沟通内容。
- 私密客户信息只写 CRM 表，不写 workspace 文档。
- **跨表更新规则：** 同步更新线索的「预计到店时间」时，仅当销售明确提供了时间才执行，不能猜时间。
